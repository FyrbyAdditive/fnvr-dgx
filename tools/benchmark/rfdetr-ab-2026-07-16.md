# RF-DETR vs YOLO26x — fleet A/B, 2026-07-16 (shodan, DGX Spark GB10)

Live 7-camera fleet (5 inference incl. one 4K H.265 + panorama; 2
record-only), ANPR + face-ID enabled, evening/night conditions.
Method: capture 10×1 s `nvidia-smi dmon -s um` + probe heartbeats +
`detections` table rates under yolo26x fp16, flip
`detector.model_family` → `rfdetr` (base, 560×560, static b1 fp16
engine), restart pipeline, re-capture after 4 min steady state.

## GPU load

| | yolo26x fp16 (640) | rfdetr-base fp16 (560, b1) |
|---|---|---|
| SM util (10×1 s) | 59–96 %, avg ≈ 78 % | 22–28 %, avg ≈ 24 % |
| Mem util column | 8–16 % | 9–16 % |

**≈ 3× SM headroom.** Note rfdetr runs a STATIC batch-1 engine, so
groups infer member-by-member — and still comes out 3× cheaper than
yolo26x with its dynamic b8 engine. FP8 quant should widen this.

## Detection behaviour

- yolo26x @ night: face+object-0 (house-side pair) published **zero**
  objects over 5 min; all-0 ≈ 0.02 obj/frame.
- rfdetr @ night, same scenes: house-side sees the parked cars
  (conf ≈ 0.42), all-0 ≈ 0.5 obj/frame, makershop sees its monitors
  ("tv" ×3, conf 0.43–0.71) and chair. Bboxes verified normalised and
  positionally sane. Recall is strictly better; nothing yolo26x found
  was lost.
- Cost of that recall at threshold 0.4: ~50× the detections-table row
  rate from threshold-hugging static scene objects (tv/chair/parked
  car re-published every frame, plus night-time "bird" noise at 0.40–
  0.48 on the 4K panorama). → pre-cluster-threshold raised 0.4 → 0.5
  in rfdetr.txt.template; per-camera class mutes remain the designed
  tool for uninteresting static classes (e.g. mute "tv" on
  cam-makershop).

## Stability

Zero CUDA errors, zero group events, push legs at full rate
(238 pkts/12 s spot-check) throughout. Engine deserialised from cache
in ~3 s per worker.

## Decision

**RF-DETR (base) is now the default detector family** (settings
default + entrypoint fallback flipped in the same commit). yolo26
stays available as `model_family=yolo26` fallback. Follow-ups: FP8
quant A/B (tools/quant-onnx), daylight recall spot-check, consider
rfdetr-medium if headroom stays this generous at 16 cameras.

## FP8 PTQ A/B (same day)

modelopt 0.43 onnx_ptq fp8 (321 Q/DQ pairs) on rfdetr-base, calibrated
with 107 real fleet frames (day+night, 5 cameras, 560×560 raw-domain).
trtexec b1 on GB10, fleet running alongside:

| | fp16 | fp8 (QDQ, --stronglyTyped) |
|---|---|---|
| median GPU compute | 3.02 ms | 2.60 ms (−14 %) |
| throughput | 266 qps | 315 qps (+18 %) |

Output parity on 6 held-out fleet frames (top boxes ≥ 0.3, fp16 vs
fp8 via trtexec --loadInputs/--exportOutput): matched-box IoU
mean 0.70 / min 0.38, |Δscore| mean 0.19 / max 0.54. **FAIL** — the
quantized model moves boxes and swings scores far beyond threshold
noise (DETR decoder sensitivity; naive whole-graph PTQ).

**Verdict: fp16 stays the RF-DETR default.** With ~3× SM headroom at
7 cameras there is no capacity pressure. Future path if needed:
selective quantization (`--nodes_to_exclude` on decoder/attention,
quantize backbone only) or NVFP4 with QAT. Artifacts kept under the
fnvr-data volume: nvinfer-dbg/quant-out/.

## Dynamic-batch attempt (2026-07-17): blocked upstream

Two approaches, both dead ends, documented so nobody re-treads:
1. ONNX surgery (input/output dim0 → symbolic): TRT builds but folds
   the engine back to static [1,…] — the exported graph hardcodes
   batch=1 in Reshape targets.
2. torch re-export with dynamic_axes (monkeypatching
   rfdetr.deploy.export.export_onnx): tracer fails with
   SymbolicValueError in rfdetr/models/backbone/projector.py:44 —
   the projector LayerNorm constructs shapes from runtime dims, which
   is untraceable with a symbolic batch dim.

A static b8 export would pad every partial batch to 8 (nvinfer static
engines infer the full batch) — a regression for solo/small groups.
Verdict: groups keep sequential b1 inference; cross-camera batching
is deferred to the Triton (nvinferserver) refactor, or an upstream
rfdetr fix to the projector. SM headroom (20–45 % at 7 cameras) makes
this a non-blocker.
